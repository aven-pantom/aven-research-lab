# Rejected Path — AI App Builder as the Main Category

## Why it initially seemed promising

The obvious interpretation of generative AI plus end-user programming is:

```text
user describes app → AI generates app → user publishes app
```

This is attractive because it is easy to explain and easy to demo.

## What weakens it

Most useful software is not a one-shot artifact.

It has:

- changing data,
- edge cases,
- permissions,
- migrations,
- UI state,
- user habits,
- workflows that drift,
- maintenance needs,
- trust requirements.

A generated app that cannot be inspected, repaired, migrated, or adapted becomes another brittle artifact.

## Failed assumption

The failed assumption is:

```text
the hardest part is initial app generation
```

The stronger assumption is:

```text
the hardest part is keeping a situated tool fitted to changing work
```

## Useful remnant

Prompt-to-app remains useful as the first fitting.

But the category should not be framed around generation alone. The durable category is:

```text
agent-maintained personal software workshops
```

## Reconsider if

Reconsider the pure app-builder framing if:

1. generated apps become reliably maintainable without manifests,
2. users show low need for iterative fitting,
3. deployment and auth become trivial,
4. schema migration stops being a problem,
5. customers value app quantity over fit and durability.

Until then, treat “AI app builder” as a demo surface, not the product center.
